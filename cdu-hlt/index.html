知識樹與課程地圖

本課程把原始 commissioning plan 拆成「設計基準 → 前置條件 → 系統介面 → 儀器文件 → 測試腳本 → 驗證判定 → 交付文件 → AI 算力中心應用」八層知識樹。學員不是只背程序，而是要能判斷測試是否有效、證據是否完整、風險是否可控。

課程知識樹

  • HLT / IST 試運轉主線
    • 測試目的：容量、冗餘、熱分佈、弱點識別
    • 設計基準：340 kW、51,000 CFM、冷通道 20~25°C、RH 30~70%
    • 系統邊界：Electrical、CRAH/P2100、XDU/CDU、PG25、BMS、Fire
    • 測試方法：0/25/50/75/100% 階梯負載、通訊中斷、單機故障、漏水模擬、TA Valve 流量設定
    • 證據鏈：BMS trend、data logger、熱影像、截圖、事件告警、每日下載
    • 交付：測試腳本、記錄表、趨勢資料、偏差清單、復歸確認、最終報告

原文文件解構

原文件定位為 HLT Requirement and Method Statement，核心不是單一設備測試，而是用熱負載與故障情境證明 Lab Room 在設計最大負載下仍能維持環境條件，並能在設備失效或通訊異常後復歸。

文件目的

為高雄專案 Lab Room HLT 與整合系統測試建立方法聲明、測試活動理由、人員角色、觀察點與故障情境。

測試哲學

每個 scenario 後要驗證 effect、reset system、再次驗證 effect，形成「故障注入 → 觀察 → 復歸 → 再確認」閉環。

干預原則

除非嚴重狀況或依腳本復歸，測試中任何人不得干預系統運轉；技術變更需共同協議。

設計基準與系統邊界

設計基準是 HLT 是否有效的錨點。沒有設計負載、環境條件、冗餘條件與測點資料，就無法判斷測試結果是否合格。

負載模型

Lab Room 模擬 340 kW IT load，應確認假負載分布能代表機櫃區發熱，而不是集中在單點。

風量模型

340 kW × 150 CFM/kW = 51,000 CFM；6 台運轉時每台 P2100 目標約 8,500 CFM。

冷通道條件

冷通道溫度 20~25°C；相對濕度 30~70%，且 IT load 至少達 50% 才能驗證濕度控制有效性。

水側基準

CRAH total water flow 482.4 LPM、進水 7°C；可轉為每台 80.4 LPM 的查核點。

工程解讀：HLT 不是「把電熱器打開」而已。真正的設計基準包含負載分布、冷熱通道隔離、風量、冷凍水流量、BMS 趨勢、冗餘機台策略與測試期間不干預原則。

前置條件與測試環境準備

前置條件是 HLT 成敗最大分水嶺。若系統仍有未結缺失、告警未清、儀器未校準、資料未開始記錄，後續測試即使看似通過，也缺乏可追溯性。

系統就緒

  • 所有獨立系統 commissioning record 已完成並由 Cx Manager / Consultant / Client team 審核。
  • 所有系統在核准 setpoint 且無 alarm 下運轉。
  • 假負載安裝於指定機櫃區，並平衡各相與各電源來源。

量測就緒

  • 溫濕度 logger 安裝於測試區合適位置。
  • 測試前 1 小時開始記錄，間隔 1 分鐘。
  • Logger 編號需標示於平面圖上。
  • 熱影像需涵蓋冷通道與熱通道指定位置。

場域就緒

  • 臨時熱通道 containment 於 HLT 前完成。
  • Command Centre 設於 BMS workstation 位置。
  • 對講機充電、角色定位、緊急口令完成確認。

測試方法拆解

容量測試

以假負載分階段達到 0/25/50/75/100%。每階段持續足夠時間，觀察冷通道溫度、回風溫度、CRAH fan speed、冷凍水供回水與總功率。

通訊中斷

停用 CRAH / P2100 network communication，確認本地控制器維持最後已知 setpoint，BMS 產生告警，復歸後告警清除且恢復自動。

單機故障

逐台 P2100/CRAH 故障並復歸，驗證剩餘機台自動加載、維持冷卻設計容量，不產生不可接受熱點。

流量設定

TA energy valve 50% loading 設 86.3 LPM，100% loading 設 172.5 LPM；需截圖並核對 BMS control point 映射。

感測校準

TT/GT 溫濕度與 dew point sensor 需用參考儀器或受控環境比對，截圖保存手持儀與 BMS 畫面。

漏水保護

對 Rack / NRack leak sensor 施加導電液，確認 shutoff valve 關閉、warning light / buzzer 啟動、BMS alarm 正確、復歸可用。

驗證重點與接受準則

驗證重點應從「設備反應」提升到「系統能力」：是否能承受負載、是否能維持環境、是否能在故障後自動調節、是否能完整復歸、是否有可追溯證據。

環境性能

  • 冷通道 20~25°C。
  • RH 30~70%，且至少 50% IT load 才具代表性。
  • 熱點不得持續擴大；熱影像應無明顯短路回風或漏風。

冷卻容量

  • 100% load 下仍可克服最大冷卻需求。
  • P2100 每台約 8,500 CFM。
  • 每台水流不低於 80.4 LPM；總流量對應 482.4 LPM。

冗餘韌性

  • 單台 CRAH/P2100 故障後剩餘機台 ramp up。
  • 通訊中斷後本地控制維持最後 setpoint。
  • 復歸後無 failed point / alarm 殘留。
重要補強：原文已有測試步驟，但正式專案建議補入明確 pass/fail limit，例如穩定時間、允許溫度偏差、告警反應時間、復歸時間、流量容許誤差、截圖命名規則與資料下載路徑。

測試期間不允許任意手動介入，否則會破壞因果鏈。例外為安全風險或指揮官指令。

若 abort，控制室透過無線電宣布，並由 UPS vendor 盡可能隔離假負載供電。

交付文件解析

交付文件應支援三件事：測前能證明 readiness，測中能重建事件，測後能判斷 pass/fail 與偏差關閉。

測前交付

  • HLT method statement / script
  • 電力、機械、消防、BMS A3 圖
  • 設備 startup / pre-functional record
  • 儀器校驗與 logger layout
  • 風險評估與 emergency protocol

測中交付

  • 逐步測試記錄表
  • BMS screenshot before/after each test
  • 1-minute trend data
  • 熱影像與現場照片
  • 事件/告警歷史 log

測後交付

  • HLT final report
  • Pass/fail summary
  • 偏差與 punch list
  • 重測與關閉證據
  • O&M baseline 與維運監測建議
交付文件解析：文件不是附件堆疊，而是證據鏈。每一筆「測試步驟」都應能連到「時間戳」、「設備狀態」、「趨勢曲線」、「截圖」、「現場觀察」、「負責人簽名」與「是否復歸」。

AI 算力中心案例：液冷 CDU + 空冷 CRAH 混合驗證

假設一座 20 MW AI 算力中心，GPU rack 採直接液冷，CDU/XDU 負責二次側冷卻液分配，CRAH 負責殘餘空冷與房間環境控制。HLT 不只驗證溫度，更驗證「液冷支路故障時，白區是否仍能安全過渡」。

案例設定

負載

GPU Rack 8~16 櫃，每櫃 80~120 kW；以假負載或液冷 load emulator 模擬二次側熱量。

冷卻

CDU/XDU 供應 PG25 或其他冷卻液；CRAH 6+1 保持白區冷通道與殘餘熱負載。

風險

高熱密度下，任何水流不足、閥位錯誤、漏水誤動作或 BMS 點位錯，都會迅速轉化為 SLA 風險。

痛點、流程、成本與效益

HLT 的成本不是「一天測試人力」，而是降低未來正式上線後熱失控、漏水、誤告警、不可復歸與 SLA 中斷的保險費。

痛點 1：測試像表演。
只看瞬間截圖，不保留完整趨勢，導致無法證明熱穩定與復歸。
痛點 2：假負載不像 IT。
負載集中、相位不平衡、冷熱通道未封閉，結果不代表真實上線。
痛點 3：BMS 點位不可信。
感測器未校準、點位 mapping 錯、告警反相，導致營運判斷失真。

知識點複習與實作測驗

完成本教案後，學員應能用測試邏輯解讀 commissioning plan，而不是只照表操作。

重要知識點

  1. HLT 的四個目的：容量、冗餘、熱分佈、弱點識別。
  2. Lab Room 設計負載為 340 kW，推導風量需求 51,000 CFM。
  3. 冷通道溫度 20~25°C，RH 30~70%，且 IT load 至少 50%。
  4. Data logger 應測前 1 小時啟動，1 分鐘間隔記錄。
  5. 測試期間除安全或指揮官指令，不得任意干預。
  6. 每個 scenario 後需 reset 並確認 restored normal operation。
  7. BMS trend、截圖、event log 與測試表需形成完整證據鏈。
